# S-DNA Update & GitHub Push Governance (L1/L2/L3)

목표: S‑DNA가 새겨진 스킬/레포를 **꾸준히 개선**하되,
- 보안 감사 기준을 훼손하지 않고
- 푸시는 L2/L3 승인 게이트를 통과하며
- 허위/취약 코드 삽입을 누적 경고로 관리하고
- 실효 없는 산출물은 단종/변경한다.

---

## 0) 정의
- **S‑DNA Asset**: S‑DNA 헤더/manifest/report 표준이 적용된 스킬 또는 GitHub 레포.
- **Update**: 문서/코드/설정/리포트 스키마/런북/증빙 링크 변경.
- **Push**: 원격 GitHub(공개/비공개 포함)에 커밋을 푸시.

---

## 1) 역할 & 권한 (3-Tier)
### L1 (자율)
- 가능: 로컬 변경, 실험, DRY_RUN, 문서 초안 작성, 이슈/PR 초안.
- 불가: 원격 push(기본 금지). 공개 링크/제출 텍스트 변경은 L2 이상.

### L2 (참모진 승인)
- 원칙: **청검 보안 veto + 청령 효용 OK**가 모두 만족되어야 원격 push 승인.
- 가능: 보안 게이트 통과 시 **원격 push 승인**.
- 책임: 변경의 안전성/재현성/증빙 일관성 + “지금 도움이 되는가”를 함께 확인.

### L3 (의장 승인)
- 필요:
  - 외부 공개 효과가 큰 변경(브랜딩/핵심 컨셉/토큰/정책)
  - 신규 외부 서비스 연결(OAuth/API Key 요구)
  - 지갑/자금 흐름에 영향을 주는 실행 로직 변경
  - 라이선스/로열티/법적 리스크가 있는 변경

---

## 2) 업데이트 원칙
1) **불편함/마찰이 발견되면 개선한다**
2) 단, **감사 기준(보안/증빙/재현성)을 훼손하는 개선은 불허**
3) 개선은 “패턴만 취하고 우리 룰로 재작성”이 원칙
4) 모든 update는 최소한 아래를 포함:
   - 무엇을 바꿨는지
   - 왜 바꿨는지(불편/버그/리스크)
   - 어떻게 검증했는지(커맨드/리포트/tx 링크)

---

## 3) Push Gate (원격 푸시 승인 체크리스트)
원격 push 전 **모두 통과**:
- [ ] Secrets 누출 없음 (`.env`, 키, 토큰, 지갑 시드/프키)
- [ ] S‑DNA 표준 유지(헤더/`reports/run_*.json`/`manifest.json`)
- [ ] 재현 커맨드 1~3개로 검증 가능
- [ ] 위험 기능 추가 없음(프론트런/MEV 공격형/무인 자동거래 기본값 등)
- [ ] 변경이 사용자/심사자에게 실질적 가치가 있음

### 문서-only 변경 라이트 게이트
- README/링크/설명 문구 같은 **문서-only** 변경은 L2에서 라이트 체크로 빠르게 승인 가능.
- 단, 보안/윤리 포지셔닝을 흐리는 문구 변경은 라이트 대상에서 제외(정식 게이트 적용).

---

## 4) 보안 금지 조항 (즉시 거부)
- 프라이빗키/시드/세션/쿠키/비밀 파라미터가 repo에 남는 모든 변경
- unlimited approval 기본값
- 사용자 확인 없이 LIVE 실행되도록 만드는 변경
- “수익 보장”/“MEV”/“스나이핑” 등 규정·윤리 리스크를 키우는 포지셔닝

---

## 5) 누적 경고 제도 (Strike System)
- **Strike 1 (경고)**: 경미한 실수(문서 오류, 링크 오류, 경미한 코드 버그) — 즉시 수정 + 재발 방지 메모
- **Strike 2 (심각 경고)**: 허위/오해 유발 정보, 안전장치 누락, 검증 없는 푸시 시도 — L2 리뷰 의무화
- **Strike 3 (중대)**: 보안 흠집(비밀 노출/백도어/자금 위험) 또는 반복적 허위 삽입
  - 조치: **자동 단종이 아니라, 즉시 동결(merge/push 금지) + L3 판정(단종/수정/복구 결정)**

※ Strike는 정치/감정용이 아니라, **감사/보안 리스크 관리**를 위한 운영 장치다.

---

## 6) 단종/변경(Deprecation) 규칙
다음 조건이면 단종/변경 후보:
- 현 시점 목표에 **전혀 도움되지 않음**
- 유지비가 과도(복잡도↑, 실패↑)
- 보안 리스크가 지속

처리 절차:
1) `DEPRECATED` 라벨 + README/문서에 안내
2) 대체 경로 제시(새 모듈/새 스킬)
3) 필요 시 아카이브 브랜치로 이동

---

## 7) 운영 메모
- 푸시는 반드시 **L2/L3 승인 게이트**를 통과한다.
- 우리는 “프로그램 자체가 감사 기준”이므로, 작은 편의보다 **감사 가능성**이 우선.
